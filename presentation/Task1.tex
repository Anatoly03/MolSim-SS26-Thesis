\simpleframe\begin{frame}[fragile]{Enums and Variants}{Definition}
    % quote: https://ocaml.org/docs/basic-data-types#variants
    % quote: https://en.cppreference.com/cpp/language/enum

    Rust uses OCaml-like enums which are more powerful than C++ enums in that every variant can branch with data.

    \vspace{1em}
    \begin{columns}[T,onlytextwidth]
        \begin{column}{0.48\textwidth}
            \begin{lstlisting}[aboveskip=0pt,belowskip=0pt,breaklines=false]
pub enum Option<T> {
    Some(T),
    None,
}

let mut a
    = Some(7);
a = None;
            \end{lstlisting}
        \end{column}
        \begin{column}{0.48\textwidth}
            \begin{lstlisting}[aboveskip=0pt,belowskip=0pt,breaklines=false]
pub enum Result<T, E> {
    Ok(T),
    Err(E),
}

let mut c: Result<(), String>
    = Ok(());
c = Err("troubles".to_string());
            \end{lstlisting}
        \end{column}
        
    \end{columns}
    \vspace{0.5em}
\end{frame}

\simpleframe\begin{frame}[fragile]{Enums and Variants}{Pattern Matching}
    This feature becomes powerful with pattern matching, which allows Rust to encode `branching' into the data type, setting a design choice for error handling.

    \vspace{0.5em}
    \begin{columns}[T,onlytextwidth]
        \begin{column}{0.48\textwidth}
            \begin{lstlisting}[aboveskip=0pt,belowskip=0pt,breaklines=false]
let value = Some(7);

let exists = match value {
    Some(_) => true,
    None => false,
}

// exists == true
            \end{lstlisting}
        \end{column}
        \begin{column}{0.48\textwidth}
            \begin{lstlisting}[aboveskip=0pt,belowskip=0pt,breaklines=false]
let hello = fs::open("hello.txt");

let file = match hello {
    Ok(f) => f,
    Err(e) => return Err(e),
}

// either `file' is here or exit
            \end{lstlisting}
        \end{column}
    \end{columns}
\end{frame}

\simpleframe\begin{frame}{Error Handling}
    \begin{tabular}{ m{0.46\textwidth} | m{0.46\textwidth} } 
        \textbf{Rust} & \textbf{C++} \\
        \hline
        \texttt{Result<T, E>} & \texttt{try} and \texttt{catch} \\
        \texttt{Option<T>} & \texttt{std::optional<T>} \\
        \hline
    \end{tabular}
    \vspace{1em}

    Rust does not have stack-unwinding exceptions which can be `caught' and handled at runtime. Instead, there are two types of error handling possibilities in Rust:

    \vspace{1em}
    \begin{tabular}{ m{0.46\textwidth} | m{0.46\textwidth} } 
        \hline
        \textbf{Recoverable} & \textbf{Unrecoverable} \\
        \hline
        \vspace{0.5em}
        \texttt{fn handle() -> Result<T, E>} &
        \vspace{0.5em}
        \texttt{panic \!("this is very very bad");} \\
        % {\begin{itemize}
        %     \item \texttt{Ok(T)}: the operation succeeded and returned a value of type \texttt{T}.
        %     \item \texttt{Err(E)}: the operation failed and returned an error of type \texttt{E}.
        % \end{itemize}}
        \hline
    \end{tabular}
    \vspace{1em}

    If a program encounters an \blue{unrecoverable error}, you \blue{panic}. This is equivalent to throwing an error which unwinds back to the main function, only calling destructor logic on the way.
\end{frame}

    % \vspace{1em}
    % \begin{itemize}
    %     \item Only owned values can be dropped. \gray{Intuitively, you cannot destroy items you lend from the library.}
    %     \item When an owned value goes out of scope, it is automatically dropped. \gray{Intuitively, if you lose a pen, it is gone.}

    \vspace{1em}
    \begin{itemize}
        \item Each value in Rust has an owner.
        \item There can only be one owner at a time.
        \item When the owner goes out of scope, the value will be dropped.\footcite{rust-ownership}
    \end{itemize}

    \vspace{0.5em}
    \gray{Opened files (and streams in general) close in the destructor logic making it impossible to forget to close files (or streams).}
\end{frame}

\nofooterframe\begin{frame}[t, fragile]{Rust Ownership Model}{The so-to-say 'first big boss' to beginners}
    \centering
    \begin{tikzpicture}[node distance=1cm and 1cm, minimum height=1cm, minimum width=3cm]
        \node[draw, rectangle] (owned) {Owned};
        \node[draw, rectangle, right=of owned] (mutborrow) {\texttt{\&mut}};
        \node[draw, rectangle, gray!30, right=of mutborrow] (borrow) {\gray{\texttt{\&}}};

        \draw[->, thick] (owned) -- (mutborrow) node[midway, above] {};
        \draw[->, gray!30, thick] (mutborrow) -- (borrow) node[midway, above] {};
        \draw[->, gray!30, thick] (mutborrow) to[bend left = 10] (borrow) node[midway, above] {};
        \draw[->, gray!30, thick] (mutborrow) to[bend right = 10] (borrow) node[midway, above] {};
        \draw[->, gray!30, thick] (owned) to[bend left = 15] (borrow) node[midway, above] {};
    \end{tikzpicture}

    \vspace{1em}
    \begin{itemize}
        \item You can take a reference mutably of a value.
        \item There can only be one mutable reference at a time. A mutable reference blocks other references to the value.
    \end{itemize}
    
    \vspace{0.5em}
    \begin{columns}[T,onlytextwidth]
        \begin{column}{0.48\textwidth}
            \centering\textbf{Not Allowed}
            \begin{lstlisting}[aboveskip=0pt,belowskip=0pt,breaklines=false]
let mut s = String::from("Hi!");

let r1 = &mut s; // &mut s
let r2 = &mut s; // &mut s

println!("{r1} {r2}"); // &r1
            \end{lstlisting}
        \end{column}
        \begin{column}{0.48\textwidth}
            \centering\textbf{Allowed}
            \begin{lstlisting}[aboveskip=0pt,belowskip=0pt,breaklines=false]
let mut s = String::from("Hi!");

let mut r1 = &mut s; // &mut s
let r2 = &mut r1; // &mut r1

println!("{r2} {r2}"); // &r1, &r2
            \end{lstlisting}
        \end{column}
    \end{columns}
\end{frame}

\simpleframe\begin{frame}[t, fragile]{Rust Ownership Model}{The so-to-say 'first big boss' to beginners}
    \begin{center}
        \begin{tikzpicture}[node distance=1cm and 1cm, minimum height=1cm, minimum width=3cm]
            \node[draw, rectangle] (owned) {Owned};
            \node[draw, rectangle, right=of owned] (mutborrow) {\texttt{\&mut}};
            \node[draw, rectangle, right=of mutborrow] (borrow) {\texttt{\&}};
    
            \draw[->, thick] (owned) -- (mutborrow) node[midway, above] {};
            \draw[->, thick] (mutborrow) -- (borrow) node[midway, above] {};
            \draw[->, thick] (mutborrow) to[bend left = 10] (borrow) node[midway, above] {};
            \draw[->, thick] (mutborrow) to[bend right = 10] (borrow) node[midway, above] {};
            \draw[->, thick] (owned) to[bend left = 15] (borrow) node[midway, above] {};
        \end{tikzpicture}
    \end{center}

    \begin{itemize}
        \item You can take multiple immutable references of a value.
        \item While there is at least one immutable reference, the value cannot be mutated.
    \end{itemize}

    \vspace{1em}
    \blue{Intuition:} If you own a book, you can lend it (\blue{borrow}), write to it (\blue{mutate}), or throw it away (\blue{drop}).
    If you lend it to someone, you usually do not want the person to write in the book or throw the book away.
    
    But if you are fine with it, you \emph{explicitly} tell the person that they can write in the book (\blue{mutable borrow}).
    Only one person can logically write in the book, and if you lend the book, it's not going to be you.
\end{frame}

\simpleframe\begin{frame}[fragile]{Rust Ownership Model}{Cheatsheet}

    \begin{tabular}{ m{0.36\textwidth} | m{0.56\textwidth} } 
        \textbf{Syntax} & \textbf{Reading} \\
        \hline
        \texttt{a: T} & \texttt{a} is an \blue{immutable owned} value of type \texttt{T} \\
        \texttt{mut a: T} & \texttt{a} is a \blue{mutable owned} value of type \texttt{T} \\
        \texttt{a: \&T} & \texttt{a} is an \blue{immutable reference} to a value of type \texttt{T} \\
        \texttt{a: \&mut T} & \texttt{a} is a \blue{mutable reference} to a value of type \texttt{T} \\
    \end{tabular}

    % \vspace{1em}
    % \begin{tabular}{ | m{0.15\textwidth} | m{0.25\textwidth} | m{0.48\textwidth} | } 
    %     \hline
    %     \textbf{Syntax} & \textbf{Description} & \textbf{Permissions} \\
    %     \hline
    %     \texttt{a: T} & \blue{owned} & read, create immutable borrow, drop \\ \hline
    %     \texttt{mut a: T} & \blue{owned mutably} & read, write, create immutable and mutable borrows, drop \\ \hline
    %     \texttt{a: \&T} & \blue{immutable borrow} & read, create immutable references \\ \hline
    %     \texttt{a: \&mut T} & \blue{mutable borrow} & read, write, create single mutable reference, create immutable references \\ \hline
    % \end{tabular}
    
    \vspace{1em}
    Borrow types like `\texttt{mut a: \&T}' and `\texttt{mut a: \&mut T}' are also possible but less common in practice.

    \blue{Recap:} A value can have either one mutable or multiple immutable references.
    
    \blue{Note:} A reference has to be valid at any point in its' scope and cannot outlive the value it points to. Lifetimes will be explained later in detail.
\end{frame}

\nofooterframe\begin{frame}[t, fragile]{Rust Ownership Model}{Problems I've Faced (Problem)}
    I had a problem with the borrow checker when implementing pairwise particle iteration. Take a look at the signature of \texttt{Vec::get\_mut} (simplified and stripped of generics):

    \begin{lstlisting}[frame=single,breaklines=false]
/// Returns a mutable reference to a particle element.
pub fn get_mut(&mut self, index: usize)
    -> Option<&mut Particle>;
    \end{lstlisting}

    Since we need to retrieve \blue{two particle references mutably}, and the \texttt{get\_mut} signature claims the entire particle vector mutably, we run into a problem with the borrow checker.

    The solution is to split the particle vector into two non-overlapping slices.\footcite{rust-split-at-mut-unchecked}

    \begin{lstlisting}[frame=single,breaklines=false]
/// Divides one mutable Particle slice into two at an index.
pub fn split_at_mut(&mut self, middle: usize)
    -> (&mut [Particle], &mut [Particle]);
    \end{lstlisting}
\end{frame}

\simpleframe\begin{frame}[fragile]{Rust Ownership Model}{Problems I've Faced (Solution)}

    \begin{lstlisting}[frame=single,breaklines=false]
let count = particles.len();

for i in 0..count {
    // newtons third law: skip same pairs
    for j in (i + 1)..count {
        let (left, right) = particles.split_at_mut(j);

        let particle1 = &mut left[i];
        let particle2 = &mut right[0];

        // ...
    }
}
    \end{lstlisting}
\end{frame}

\simpleframe\begin{frame}{Developer Experience}
    \begin{itemize}
        \item It is definitely easier to work in a programming language in which you have more years of experience. \blue{$\rightarrow$ Bias when measuring `developer experience'}.
        \item Rust compiler prevents you from compiling code that goes against ownership model. \blue{$\rightarrow$ Not beginner-friendly and slow feedback loop, but enforces code quality in the early stage of the source code creation}.
        \item C++ has a more fragmented ecosystem: CMake, \texttt{getopt}, \texttt{doxygen}, \texttt{clang-format}, manual dependency installation like \texttt{spdlog}, ...
        \item Here: \text{cargo} for everything including dependency management, similar to \text{pip} in Python. \blue{$\rightarrow$ Easier management without less control}.
    \end{itemize}
\end{frame}
